# memlog — Implementation Plan

> Append-only, git-backed memory for agents: facts in, markdown out, with provenance ("learned this in session X"). The "memory" everyone wants, but auditable.

This document is a complete, decision-free specification. Every choice has already been made. Do not introduce alternative libraries, formats, or commands. If something is genuinely ambiguous, follow the "When in doubt" rules in §12.

---

## 1. Goals and Non-Goals

**Goals**

1. A single-binary CLI (`memlog`) that agents and humans can call to record facts, correct them, and render a readable `MEMORY.md`.
2. Every fact is immutable once written. Corrections happen by appending new entries that supersede or retract old ones — never by editing history.
3. Every fact carries provenance: who/what wrote it, in which session, when, and optionally why.
4. The store is a plain git repository. `git log` alone must be enough to audit the entire memory history.
5. Deterministic output: the same journal always renders to byte-identical markdown.

**Non-Goals (do NOT build these)**

- No vector search, embeddings, or LLM calls of any kind.
- No daemon, server, or background process. (An MCP server is a future milestone, M5, explicitly out of scope for the initial build.)
- No database. No SQLite. Plain files only.
- No automatic fact extraction from conversations. memlog only stores what it is explicitly told.
- No remote sync logic beyond what git itself provides.

## 4. The Memory Store (what `memlog init` creates)

A memory store is a directory (default `./.memlog` resolved by walking up from CWD, like git does; overridable with `--store PATH` or env `MEMLOG_STORE`). Structure:

```
.memlog/
├── journal/
│   └── 2026-06.jsonl         # one file per UTC month, append-only
├── MEMORY.md                 # rendered output (regenerated, committed)
├── meta.json                 # {"version": 1, "created_at": "...", "store_id": "<ulid>"}
└── .gitignore                # contains: *.lock
```

`memlog init`:
1. Fails with exit code 2 if a store already exists at the target.
2. Creates the directory structure above.
3. Runs `git init` **only if** the store is not already inside a git work tree (check via `git rev-parse --is-inside-work-tree`). If it is inside one, reuse the enclosing repo.
4. Writes `meta.json` and an initial empty `MEMORY.md` (rendered from zero entries).
5. Commits with message `memlog: init store <store_id>`.

## 5. Entry Schema (the core data model)

Each line in a journal file is one JSON object, UTF-8, no trailing spaces, `\n` terminated, keys in exactly this order (use a struct; Go's encoding/json preserves struct field order):

```json
{
  "id": "01J9XK7M3QJ8Z6W4V2T1R0PQNM",
  "ts": "2026-06-12T14:03:22Z",
  "op": "add",
  "fact": "The staging database password rotates every 30 days.",
  "tags": ["infra", "staging"],
  "subject": "staging-db",
  "session": "claude-code-2026-06-12-a",
  "agent": "claude-code",
  "source": "user told me directly",
  "ref": null
}
```

Field rules (enforce all in `model.Validate`; invalid entries are rejected at write time with exit code 2):

| Field | Type | Rules |
|---|---|---|
| `id` | string | ULID, generated by memlog, never user-supplied |
| `ts` | string | RFC 3339 UTC with seconds precision, generated by memlog (overridable only via hidden `--ts` flag for tests) |
| `op` | string | one of `add`, `supersede`, `retract` |
| `fact` | string | required for `add`/`supersede`; must be empty for `retract`; max 2000 chars; no control chars except none (single line — reject `\n`) |
| `tags` | []string | 0–10 items; each matches `^[a-z0-9][a-z0-9-]{0,31}$`; sorted ascending; deduplicated |
| `subject` | string | optional; same regex as a tag; groups facts in rendering |
| `session` | string | required; 1–128 chars, printable ASCII |
| `agent` | string | optional; 0–64 chars |
| `source` | string | optional free text, max 500 chars, single line |
| `ref` | string/null | required ULID of an existing prior entry for `supersede`/`retract`; must be null for `add`; the referenced entry must have op `add` or `supersede` and must not already be retracted or superseded (reject with exit 2 and message `entry <id> is already superseded by <id2>`) |

**Semantics**

- `add`: a new fact.
- `supersede`: replaces the fact in entry `ref` with this entry's `fact`. The chain add→supersede→supersede… forms the version history of one logical fact. Tags/subject of the supersede entry replace the old ones (they are not merged).
- `retract`: the logical fact whose latest version is `ref` is no longer true; it disappears from `MEMORY.md` (but appears in `memlog history`).

**Resolution algorithm** (used by render, show, search): read all journal files in filename order, then line order. Build a map from entry id → entry. A logical fact's "head" is the last entry in its supersede chain. A fact is *live* iff its head is not the target of any `retract`. Because the journal is strictly append-only and ULIDs are time-ordered, a single forward pass suffices: track `replacedBy[ref] = id` and `retracted[ref] = id`.

## 6. Write Path (must be exactly this)

Every mutating command (`add`, `supersede`, `retract`) does:

1. Acquire `journal.lock` (flock) in the store dir; wait up to 5s, then fail with exit 3 and message `store is locked by another process`.
2. Validate the entry (§5).
3. Append the JSON line to `journal/<YYYY-MM>.jsonl` (current UTC month; create file if absent). Use `O_APPEND`. fsync the file.
4. Re-render `MEMORY.md` (§8) and write it atomically (write temp file in store dir, rename).
5. `git add` the journal file + `MEMORY.md`; `git commit` with the message format in §7.
6. Release the lock.
7. Print the new entry id to stdout (just the ULID, nothing else) and exit 0.

If git commit fails (e.g., nothing staged — impossible in practice, or git missing), exit 4 with the git stderr. The journal append has already happened and stays; the next successful command's commit will pick it up. Document this in README under "crash recovery", and implement `memlog doctor` to detect/fix it (§9).

## 7. Git Conventions

- Commit message, first line: `memlog: <op> <id>` e.g. `memlog: add 01J9XK7M3QJ8Z6W4V2T1R0PQNM`.
- Commit body (separated by blank line): the fact text (or for retract: `retracts <ref>`), then a trailer block:
  ```
  Memlog-Session: claude-code-2026-06-12-a
  Memlog-Agent: claude-code
  ```
- Author: do not override; use the user's git config.
- Never use `git commit --amend`, never rebase, never force-push, never delete journal lines. The tool must not expose any history-rewriting capability.

## 8. Rendering `MEMORY.md` (deterministic)

Render from the resolved live facts only. Exact format:

```markdown
# Memory

_Last updated: 2026-06-12T14:03:22Z · 42 live facts · 7 retracted · store 01H..._

## staging-db

- The staging database password rotates every 30 days. `#infra #staging` ⟨[01J9XK7M](#provenance) · session claude-code-2026-06-12-a⟩

## (no subject)

- ...

## Provenance

| id | learned | session | agent | source | history |
|---|---|---|---|---|---|
| 01J9XK7M | 2026-06-12 | claude-code-2026-06-12-a | claude-code | user told me directly | v1 |
```

Rules:

1. Sections: one `##` per subject, subjects sorted ascending; facts without subject go last under `## (no subject)` (omit the section if empty).
2. Within a section, facts sorted by head-entry ULID ascending.
3. Each bullet: fact text, then tags as `` `#tag` `` space-separated (omit if none), then `⟨first-8-chars-of-head-id · session <session-of-head>⟩`.
4. Provenance table lists every live fact's head entry; `history` column shows `v<N>` where N = chain length; `learned` is the date (YYYY-MM-DD) of the **original** add entry in the chain.
5. Timestamps in the header line come from the newest entry in the journal (any op), not wall-clock time — this is what makes rendering deterministic.
6. End file with exactly one trailing newline.

Golden-file tests (§10) lock this format down.

## 9. CLI Specification

Global flags: `--store PATH`, `--json` (machine-readable output where noted). Exit codes: 0 ok, 1 not found/empty result, 2 validation/usage error, 3 lock timeout, 4 git/system error.

| Command | Synopsis | Behavior |
|---|---|---|
| `memlog init [PATH]` | create store | §4 |
| `memlog add FACT --session S [--tags a,b] [--subject x] [--agent A] [--source TEXT]` | append fact | §6; prints new ULID |
| `memlog supersede REF FACT --session S [...same flags]` | new version of fact REF | REF may be an unambiguous id prefix (≥8 chars); ambiguous prefix → exit 2 listing candidates |
| `memlog retract REF --session S [--source TEXT]` | retract fact | |
| `memlog show REF` | print one logical fact | human format: current text, status (live/retracted), full chain oldest→newest with ts/session; `--json` prints the raw entries array |
| `memlog search QUERY [--tag T] [--subject X] [--all]` | case-insensitive substring search over live facts (`--all` includes retracted/superseded text) | one line per hit: `<id8>  <subject or ->  <fact>`; exit 1 if no hits |
| `memlog history` | full journal, oldest first | one line per entry: `<ts>  <op>  <id8>  <fact-or-ref>` |
| `memlog render` | force re-render MEMORY.md + commit if changed | prints `unchanged` or `updated` |
| `memlog sessions` | list distinct sessions with entry counts, newest first | |
| `memlog doctor` | integrity check | verifies: every line parses, ids unique, refs resolve, no ref to retracted/superseded entries, MEMORY.md matches a fresh render, no uncommitted store changes. Reports each problem; `--fix` re-renders and commits uncommitted journal state. Exit 0 clean / 1 problems found |

All human-readable errors go to stderr. `--json` outputs must be stable (documented in README).

## 10. Testing Requirements (CI must enforce all)

1. **Unit**: `model.Validate` — table-driven tests covering every rule in §5 (valid + at least one invalid case per rule).
2. **Store**: append + resolution algorithm — chains of add/supersede/retract, including superseding a superseded entry (must fail), retracting a retracted entry (must fail), ref to nonexistent id (must fail).
3. **Golden render tests**: ≥4 fixtures in `testdata/` (empty store; subjects+no-subject mix; supersede chains; retractions). Compare byte-for-byte.
4. **CLI integration**: build the binary in a temp dir with a real `git init`-ed store; run the full lifecycle init→add×3→supersede→retract→search→show→doctor; assert exit codes, stdout, and that `git log --oneline` has the expected count and messages.
5. **Concurrency**: two processes adding simultaneously — second blocks or fails with exit 3, journal never interleaves/corrupts.
6. **Determinism**: render twice, assert identical bytes; clone the repo elsewhere, `memlog render` prints `unchanged`.
7. Month rollover: entries written with hidden `--ts` in two different months land in two journal files and resolve correctly.

## 11. Milestones (implement strictly in order; each ends with green CI)

- **M1 — Core store**: model, validation, journal append, locking, resolution algorithm, `init`, `add`, `history`. Tests 1, 2, 5, 7.
- **M2 — Lifecycle**: `supersede`, `retract`, `show`, prefix resolution. Extend tests.
- **M3 — Render + search**: renderer, `render`, `search`, `sessions`, golden tests, determinism tests.
- **M4 — Robustness**: `doctor`, crash-recovery path, `--json` outputs, README with full command docs + an "agent usage" section showing how a coding agent should call memlog (one-line examples for add/supersede/retract).
- **M5 (out of scope now, document as future work)**: MCP server exposing add/search/show as tools.

## 12. When in Doubt

1. Prefer rejecting input (exit 2, clear message) over guessing.
2. Never mutate or delete an existing journal line or rewrite git history, under any circumstance, including in `doctor --fix`.
3. Keep stdout machine-friendly (ids, results); everything chatty goes to stderr.
4. If a needed behavior is truly unspecified here, choose the option that keeps the journal append-only and rendering deterministic, and note the decision in README under "Design notes".
